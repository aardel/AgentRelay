# AgentRelay Task Queue

The task queue provides durable, observable tracking for work routed between machines. Every `agent-send` call to a remote peer creates task records on both sides, with status synchronized via relay callbacks.

## Storage

SQLite database at `~/.config/agentrelay/tasks.db`, WAL mode, asyncio-safe via `run_in_executor` + `asyncio.Lock`.

## Status Lifecycle

```
Originating machine:   queued → sent → [acked] → completed / failed
Receiving machine:     received → running(session_id) → completed / failed
```

State transitions are enforced — invalid transitions are rejected.

## Key Fields

| Field | Description |
|-------|-------------|
| `task_id` | UUID generated by the originator; shared with the receiver as the join key |
| `target_node` | Destination node name (matches peer registry and `ssh_hosts.json`) |
| `target_agent` | Agent identifier (`claude-interactive`, `codex`, etc.) |
| `source_node` | Originating node name |
| `permission_profile` | `safe`, `project_write`, or `full_auto` |
| `session_id` | PTY session ID written by the receiver on launch; anchor for terminal attach |
| `originator_task_id` | On receiver records: the task_id from the originator |
| `reply_to` | Full URL the receiver POSTs status updates to (`http://<originator>:9876/api/tasks/{id}/status`) |
| `created_at` / `started_at` / `completed_at` | Timestamps for duration tracking |

## Two-Sided Tracking

Both the originating and receiving machine maintain a task record, each tracking their own view of the lifecycle. The originator's `task_id` is included in the `/forward` payload and used as the receiver's `originator_task_id`, making both records joinable without additional lookup.

When the receiver's PTY session closes, it POSTs a status update to `reply_to`. The originator updates its record to `completed` or `failed`.

## API Endpoints

| Endpoint | Auth | Description |
|----------|------|-------------|
| `GET /api/tasks` | localhost-only | List tasks (filterable by status, node, limit) |
| `GET /api/tasks/{id}` | localhost-only | Get a single task record |
| `POST /api/tasks/{id}/status` | token (peer-to-peer) | Receive a status update from a remote machine |

Status updates flow via push (receiver → originator). The originator never polls the receiver's `/api/tasks`.

## Remote Terminal Attach

When a receiver task has a `session_id`, the terminal is reachable at:

```
ws://<target_node>:9876/terminal/{session_id}
```

The Tasks UI panel shows an **[attach]** link for any task with a `session_id`. Clicking it opens a new terminal tab connected to that session, live or completed.

## Live UI

The Tasks panel uses Server-Sent Events (`GET /api/tasks/events`) for real-time updates. The SSE endpoint pushes an event on every status transition; the panel fetches the updated task list immediately on each event. Polling runs only while active tasks exist and stops when all tasks reach a terminal state.
